# Uplink Wave Isolation Patch

This document describes `Patch_UplinkWaveIsolation`, the Harmony patch that changes
the behaviour of Terminal Uplink completion so that it only stops its **own** enemy
waves instead of every active warden objective wave in the level.

- **Source**: `AutogenRundown/src/Patches/Patch_UplinkWaveIsolation.cs`
- **Related**: `AutogenRundown/src/Patches/TerminalUplink.cs` (unrelated uplink-puzzle
  patches — customisation of the puzzle itself, not wave handling)

---

## Problem

Vanilla GTFO calls `WardenObjectiveManager.StopAllWardenObjectiveEnemyWaves()` when
any Terminal Uplink objective completes. That method walks
`WardenObjectiveManager.m_enemyWaveEventIDs` and calls `StopEvent()` on every
`Mastermind` event in the list.

`m_enemyWaveEventIDs` is **global**: any warden-triggered wave on the level is in it,
including:

- Alarm waves from unrelated security doors
- Error alarm waves (`Surge`, `ErrorAlarm`, etc.)
- Survival-phase waves (reactor, survival objectives)
- Any other objective's currently running waves

The net effect is that finishing a single uplink can accidentally cancel alarms
from completely different objectives running in parallel. For autogenerated
rundowns this breaks multi-alarm setups (e.g. an uplink that happens concurrently
with a reactor or with an error alarm trapping players in a zone).

## Goal

Make uplink completion surgical: stop **only** the waves that the uplink itself
spawned, and leave every other tracked wave running.

---

## Architecture Overview

The patch is a single static class, `Patch_UplinkWaveIsolation`, holding a small
amount of static state plus four Harmony patches and one helper class. It does
not read or mutate any datablocks — it only intercepts runtime calls.

```
┌─────────────────────────────────────────────────────────────────┐
│                    Static state (per level)                     │
│                                                                 │
│  UplinkWaveIds:          Dictionary<int, HashSet<ushort>>       │
│    node ID → set of wave event IDs spawned by that uplink       │
│                                                                 │
│  InterceptNextStopAll:   bool                                   │
│    arms the StopAll prefix to intercept the next call           │
│                                                                 │
│  CompletingUplinkNodeId: int?                                   │
│    node ID whose waves the intercepted call should stop         │
└─────────────────────────────────────────────────────────────────┘
                               ▲
                               │ populated / consumed by
                               │
┌──────────────────┐ ┌─────────┴────────┐ ┌──────────────────────┐
│ StartTerminal    │ │ TriggerSurvival  │ │ TerminalUplinkVerify │
│ UplinkSequence   │ │ Wave (Mastermind)│ │                      │
│                  │ │                  │ │                      │
│ Postfix:         │ │ Postfix:         │ │ Prefix+Postfix:      │
│ allocate a       │ │ if refNode is a  │ │ on final-round       │
│ tracking set for │ │ tracked uplink   │ │ correct code, arm    │
│ the uplink node  │ │ node, record the │ │ the StopAll          │
│                  │ │ eventID          │ │ intercept flag       │
└──────────────────┘ └──────────────────┘ └──────────────────────┘
                                                     │
                                                     ▼
                                   ┌────────────────────────────────────┐
                                   │ StopAllWardenObjectiveEnemyWaves   │
                                   │                                    │
                                   │ Prefix:                            │
                                   │ if armed → stop only the tracked   │
                                   │ uplink waves, return false         │
                                   │ (skip vanilla global stop).        │
                                   │ Otherwise → let vanilla run.       │
                                   └────────────────────────────────────┘
```

`LevelAPI.OnLevelCleanup` (registered in `Setup()`) resets all state between
levels.

## Patches in Detail

### 1. `StartTerminalUplinkSequence` — Postfix (tracking allocation)

**Target**: `LG_ComputerTerminalCommandInterpreter.StartTerminalUplinkSequence(bool corrupted)`

Fires when a player types `UPLINK_CONNECT` / `UPLINK_CORRUPTED_CONNECT` and the
verification sequence actually begins.

```csharp
var targetTerminal = corrupted
    ? __instance.m_terminal.CorruptedUplinkReceiver
    : __instance.m_terminal;

var nodeId = targetTerminal.SpawnNode.NodeID;
UplinkWaveIds[nodeId] = new HashSet<ushort>();
```

Two important details:

- **Corrupted uplinks** target a different terminal than the one the command was
  typed on (`CorruptedUplinkReceiver`). The tracking key must be the node of the
  terminal that actually receives the uplink, because that is the `refNode` the
  uplink wave is spawned at.
- **Node IDs, not references.** IL2CPP creates a fresh C# wrapper object each
  time you cross the managed/native boundary, so two wrapper instances for the
  same underlying `AIG_CourseNode` fail reference-equality checks. `NodeID` is a
  plain `int` and survives this round-trip, so it is used as the dictionary key
  everywhere.

### 2. `Mastermind.TriggerSurvivalWave` — Postfix (wave capture)

**Target**: All public instance overloads of `Mastermind.TriggerSurvivalWave`.
Found via `HarmonyTargetMethods` because there are multiple overloads — patching
by method reference alone causes an `Ambiguous match` at startup.

```csharp
public static IEnumerable<MethodBase> TargetMethods()
{
    return typeof(Mastermind).GetMethods(BindingFlags.Public | BindingFlags.Instance)
        .Where(m => m.Name == nameof(Mastermind.TriggerSurvivalWave))
        .Cast<MethodBase>();
}

[HarmonyPostfix]
public static void Postfix(AIG_CourseNode refNode, ushort eventID, bool __result)
{
    if (!__result || refNode == null) return;

    if (UplinkWaveIds.TryGetValue(refNode.NodeID, out var waveIds))
        waveIds.Add(eventID);
}
```

Every time the game spawns a warden-objective survival wave it calls
`TriggerSurvivalWave(refNode, eventID, …)`. The postfix runs after the wave was
successfully spawned (`__result == true`), looks at the `NodeID` of the spawn
node, and if that node matches a tracked uplink it records the new wave's
`eventID` in that uplink's set.

**This is the step that decides which waves belong to the uplink.** A wave is
considered "part of this uplink" if and only if its `refNode` matches the
uplink's `SpawnNode`. Any wave spawned at a different node — a door alarm in
another zone, an error alarm in the same level, a reactor wave, etc. — is
ignored here and will never be cancelled by the intercept.

### 3. `TerminalUplinkVerify` — Prefix + Postfix (arming the intercept)

**Target**: `LG_ComputerTerminalCommandInterpreter.TerminalUplinkVerify(string param1)`

This is called on every verify attempt — i.e. each time the player types a
verification code. The prefix captures the typed code into Harmony `__state`;
the postfix decides whether that attempt is the final one.

```csharp
[HarmonyPrefix]
public static void Prefix(string param1, out string __state)
{
    __state = param1;
}

[HarmonyPostfix]
public static void Postfix(LG_ComputerTerminalCommandInterpreter __instance, string __state)
{
    var puzzle = __instance.m_terminal?.UplinkPuzzle;
    if (puzzle == null || !puzzle.Connected || puzzle.Solved) return;

    if (!string.Equals(puzzle.CurrentRound.CorrectCode, __state,
                       StringComparison.OrdinalIgnoreCase)) return;

    var progress = puzzle.CurrentProgress;        // "X/Y"
    var parts = progress.Split('/');
    if (parts.Length != 2 || parts[0] != parts[1]) return;  // not final round

    var nodeId = __instance.m_terminal?.SpawnNode?.NodeID;
    if (nodeId.HasValue && UplinkWaveIds.ContainsKey(nodeId.Value))
    {
        InterceptNextStopAll = true;
        CompletingUplinkNodeId = nodeId.Value;
    }
}
```

Why this detection strategy rather than hooking completion directly:

> We can't patch `FireEndOfQueuCallbacks` because IL2CPP invokes the delegate
> directly without going through the patchable C# method wrapper.

Instead we detect "this keypress will cause completion" at verify time:

1. `puzzle.Connected` guards against races where the puzzle has been torn down.
2. `puzzle.Solved` is not yet `true` at verify time — it is set in the callback
   that fires *after* verification succeeds. If we see `Solved == true` here we
   have already run once for this completion; bail out.
3. The code check compares the typed code (`__state`) against
   `CurrentRound.CorrectCode`. If it doesn't match, this verify attempt will
   fail and no completion is coming.
4. The round check parses `CurrentProgress` which is formatted as `"X/Y"`. Only
   when `X == Y` is this attempt the final round.
5. The node must already be tracked (`UplinkWaveIds.ContainsKey`), otherwise the
   `StartTerminalUplinkSequence` postfix never ran and this is not an uplink we
   should interfere with.

If all checks pass, we arm the intercept.

### 4. `StopAllWardenObjectiveEnemyWaves` — Prefix (intercept)

**Target**: `WardenObjectiveManager.StopAllWardenObjectiveEnemyWaves()`

This is the prefix that does the actual isolation. Its job is to return
`false` (skip the vanilla method) when it is armed, and `true` (let vanilla run)
otherwise.

Flow:

```
┌─ not armed ──────────────────────────────────────────────────────┐
│  return true → vanilla stops everything (unchanged behaviour)   │
└─────────────────────────────────────────────────────────────────┘

┌─ armed ──────────────────────────────────────────────────────────┐
│  1. disarm immediately (InterceptNextStopAll = false,            │
│     CompletingUplinkNodeId = null) to avoid accidentally          │
│     intercepting an unrelated StopAll that arrives later          │
│                                                                  │
│  2. fetch this uplink's wave-ID set; if empty/missing,            │
│     return true so vanilla runs (effectively a no-op fallback)    │
│                                                                  │
│  3. play APEX_PUZZLE_STOP_ALARM so the audio matches vanilla      │
│                                                                  │
│  4. on non-master clients: clean up tracking and return false.    │
│     Only the master actually stops mastermind events — the        │
│     network layer propagates the stop to clients.                 │
│                                                                  │
│  5. on master:                                                    │
│       a. get the private m_enemyWaveEventIDs list via reflection  │
│       b. for each recorded wave ID:                               │
│            - Mastermind.Current.TryGetEvent(id, out event)        │
│              then event.StopEvent()                               │
│            - remove id from m_enemyWaveEventIDs                   │
│       c. remove the tracking entry for this uplink                │
│                                                                  │
│  6. return false → vanilla StopAll does NOT run                   │
└─────────────────────────────────────────────────────────────────┘
```

Key point: because we manually drive each recorded wave's `StopEvent()` and
then prune it from `m_enemyWaveEventIDs`, the global list ends up in the same
consistent state vanilla would have left it in — *except* that waves belonging
to other objectives/alarms are still present and still running.

### Reflection access to `m_enemyWaveEventIDs`

The field is `private` on `WardenObjectiveManager`. We go through it once and
cache the result:

```csharp
private static List<ushort>? _cachedEnemyWaveEventIDs = null;
private static FieldInfo?     _enemyWaveEventIDsField = null;
```

- `_enemyWaveEventIDsField` caches the `FieldInfo` itself (cheap, process
  lifetime).
- `_cachedEnemyWaveEventIDs` caches the `List<ushort>` instance on the current
  `WardenObjectiveManager`. This is invalidated on level cleanup because a new
  `WardenObjectiveManager` is created per level.

## Lifecycle

### Happy path (uplink with no other concurrent alarms)

1. Player runs `UPLINK_CONNECT` → `StartTerminalUplinkSequence` postfix creates
   `UplinkWaveIds[nodeId] = {}`.
2. Uplink spawns its alarm wave → `TriggerSurvivalWave` postfix fires with
   `refNode.NodeID == nodeId`, records `eventID` in the set.
3. Player types codes. For each attempt `TerminalUplinkVerify` postfix runs;
   on the final correct code it arms `InterceptNextStopAll`.
4. The uplink completion callback runs in game code and calls
   `StopAllWardenObjectiveEnemyWaves`.
5. Our prefix sees the flag, stops only the tracked waves, removes them from
   `m_enemyWaveEventIDs`, plays the stop sound, returns `false`. Vanilla
   `StopAll` is skipped.

### Concurrent alarm (e.g. a door alarm from another zone running at the same time)

Same as above, except step 2 happens once for the uplink wave and
*separately* for the door alarm wave. The door alarm's `refNode` is **not** the
uplink's spawn node, so its `eventID` is never added to `UplinkWaveIds`. At
step 5 we only stop the uplink's set, so the door alarm keeps running.

### Edge cases and failure modes

- **Disconnect / puzzle torn down mid-verify**: `puzzle.Connected` guards this;
  we simply don't arm the intercept. Vanilla behaviour takes over.
- **StopAll called before we arm**: the flag is `false`, prefix returns
  `true`, vanilla runs. The cost is that the uplink wave gets double-stopped
  later — harmless because `StopEvent()` on an already-stopped event is a
  no-op and the ID has already been removed from `m_enemyWaveEventIDs`.
- **StopAll called from an unrelated path after we armed but before uplink
  completion**: the intercept would fire for the wrong caller. In practice we
  have not seen this because `StopAllWardenObjectiveEnemyWaves` is only called
  on uplink/objective completion paths; still, disarming immediately inside
  the prefix limits the blast radius to a single spurious StopAll.
- **Non-master client**: only the master drives mastermind events; the prefix
  still returns `false` on clients so their local vanilla path does not also
  try to stop things. Cleanup of tracking state still happens locally.
- **Level change**: `LevelAPI.OnLevelCleanup` clears every piece of state,
  including the cached reflection handle to `m_enemyWaveEventIDs`.

## Why this shape

A few design choices worth flagging for anyone maintaining the patch:

- **Node ID as the identity key.** Any `AIG_CourseNode` comparison in IL2CPP is
  a footgun. The dictionary is keyed by `int` everywhere to sidestep that.
- **`HarmonyTargetMethods` for `TriggerSurvivalWave`.** There are multiple
  overloads on `Mastermind`. Patching a specific one by signature is brittle
  across game updates; enumerating by name catches all of them.
- **Detect completion at verify time, not at completion time.** The real
  completion callback is an IL2CPP delegate invoked natively and cannot be
  patched reliably, so we predict completion instead: "final round + correct
  code = about to complete."
- **Disarm the flag inside the intercept prefix.** Keeping the intercept armed
  longer than necessary risks catching an unrelated StopAll. We pay the cost
  of a possibly-missed intercept (which falls back to vanilla behaviour) in
  exchange for never interfering with the wrong call.
- **Reflection into `m_enemyWaveEventIDs`.** There is no public accessor. The
  field is stable across game revisions we support; if that changes, only
  `GetEnemyWaveEventIDs` has to be updated.

## Debugging

All branches log to `Plugin.Logger.LogDebug` with the prefix
`[UplinkWaveIsolation]`. Useful log lines when diagnosing issues:

- `Tracking uplink node ID: {nodeId} (zone {LocalIndex})` — uplink was started
  and is now being tracked.
- `Captured uplink wave ID: {eventID} for node ID {nodeId}` — a wave has been
  associated with this uplink.
- `Final round correct! Setting intercept flag for node {nodeId}` — final code
  accepted, intercept armed.
- `Intercepting StopAll - stopping {N} uplink waves only for node ID {nodeId}`
  — intercept fired, only uplink waves are being stopped.
- `Intercept flag set but no tracked waves for nodeId {nodeId}` — armed, but
  the tracked set was empty; falls back to vanilla. Usually means no wave was
  ever spawned for this uplink (quiet uplink, no alarm) and the intercept is a
  no-op either way.
- `Could not find m_enemyWaveEventIDs field` — reflection lookup failed; game
  version changed and `GetEnemyWaveEventIDs` needs updating.

If the symptom is **"finishing an uplink still cancels an unrelated alarm"**,
the most likely causes are, in order:

1. The intercept never armed (`Final round correct!` line missing). Check the
   verify-postfix guards: `puzzle.Solved`, code match, `CurrentProgress`
   format.
2. The intercept armed but the tracked set also contained the unrelated wave.
   This would mean the unrelated wave was spawned at the uplink's `SpawnNode`
   — unusual, worth verifying the `refNode` the game passes for that alarm.
3. Another StopAll path consumed the intercept. Look for a `Intercepting
   StopAll` line that fires suspiciously early, before the uplink completion
   callback.
